# CH02 对话循环 — Agent 的心跳

## 2026-07-02 阶段总结：TinyAgent 心跳层升级完成

这一阶段的目标不是复刻 Claude Code，而是借鉴 `CodeReference/claude-code-book/第一部分-基础篇/02-对话循环-Agent的心跳.md` 里真正适合 TinyAgent 的核心设计，把原来的一次性任务 runner 升级成可持续交互、可观察、可恢复的 Agent loop。

主要参考点：

- CH02 的异步生成器思想：Agent loop 不应该是黑盒函数，而应该持续产出事件，让 UI/TUI 实时消费。
- CH02 的 `State / Continue / Terminal` 状态转换模型：每次继续循环要有明确原因，每次终止要有明确 reason。
- CH02 的完整 turn 生命周期：请求开始、模型响应、工具调用、工具结果回填、下一轮继续。
- CH02 的上下文预处理管线思想：上下文管理应从轻量预算到硬阻断逐级处理，而不是等 API 爆掉。
- CH02 的可观测性原则：trajectory 要能解释 Agent 每一步为什么发生。

本阶段已落地能力：

- 新增/强化事件协议：
  - `stream_request_start`
  - `assistant_delta`
  - `tool_start`
  - `tool_progress`
  - `tool_end`
  - `transition`
  - `context_edit`
- `run_start` 写入 `session_id`，方便区分同一 TUI session 下的多次 run。
- 保留 `Provider.complete(...)`，新增 `Provider.stream(...)`。
- `OpenAIChatProvider` 已支持 Chat Completions 真流式输出，并聚合成最终 `ModelTurn`，因此 loop 仍能保持原有工具协议和历史消息结构。
- 新增 `CancellationToken`，让 TUI、loop、provider、tool、bash 共享取消信号。
- TUI v1 支持持续会话、实时显示模型 delta、工具生命周期、成本、trace。
- 工具执行接入 progress callback，bash 长命令可看到运行进度。
- Windows 下 bash 工具不再优先撞坏掉的 WSL launcher，会 fallback 到 PowerShell，修复了之前真实测试中 bash 不可用的问题。
- 上下文管理升级为 v2：
  - 工具结果进入下一轮前先做预算。
  - 超预算时清理旧 tool result。
  - 本地 prompt hard limit 阻断。
  - API 返回 context too long 时执行 reactive compact retry 一次。
- terminal reason 更细：
  - `aborted_streaming`
  - `aborted_tools`
  - `blocking_limit`
  - `prompt_too_long`
  - `model_error`

关键文件：

- `harness/loop.py`：Agent 心跳主循环，消费 provider stream，产出事件，处理 Continue/Terminal。
- `harness/providers/base.py`：Provider 抽象增加 `stream(...)`。
- `harness/providers/openai_chat.py`：OpenAI Chat Completions 真流式实现。
- `harness/cancel.py`：统一取消 token。

- `harness/tui.py`：TUI v1，实时展示 streaming 和工具事件。
- `harness/context.py`：上下文预算、清理、hard limit、reactive compact。
- `harness/tools/registry.py`：工具 progress/cancel 接口。
- `harness/tools/bash.py`：bash/PowerShell fallback、取消和进度。
- `tests/test_streaming_and_control.py`：新增流式、取消、上下文阻断和 reactive compact 测试。

## 2026-07-02：CH03 第二大步 - Coding 工具体系 + 评测闭环

本阶段目标是把第一大步做好的工具内核、安全反射和生命周期事件，扩展成更像 coding agent 的“双手”，并用过程型 eval 验证工具真的改善任务能力。

新增/强化能力：

- `edit_file`：精确替换已有 UTF-8 文本文件；强制先 `read_file`；默认 `old_string` 必须唯一，不唯一时要求扩大上下文或显式 `replace_all=true`。
- `write_file`：创建新文件可以直接写；覆写已有文件必须先读且文件未变化；返回 `create/update` 摘要并更新 read cache / file history。
- `glob_files`：按文件名 glob 找文件，优先 `rg --files`，稳定排序并限制结果量。
- `grep`：按内容搜索，返回 `path:line:content`，支持 include glob、上下文行、截断提示，避免模型总用 bash grep。
- `file_info` / `show_diff`：补齐 coding agent 常用的文件体检和变更复核工具。
- Bash v2：只读命令（`ls/dir/Get-ChildItem/rg/git status/git diff/git log/git show`）可标记为只读并发；写文件、包管理、测试、提交等命令保持串行/副作用路径。
- TUI / Viewer：展示 `tool_queued`、`tool_validate`、`tool_permission`、`tool_result_persisted`、`tool_context_modified` 等工具生命周期留痕。
- eval 07-09：新增 coding eval，不只验最终文件，也验 trajectory 过程事件。

新增评测任务：

- `07_precise_edit`：只修 `app.py` 中 `compute_score` 的除零问题。Scorer 要求 `read_file -> edit_file -> bash/test`，禁止用 `write_file` 覆写 `app.py`。

## 2026-07-03：TUI ProMax - OpenCode / Claude Code 风格交互升级

本阶段目标是把原来的“工具日志面板”改成更像 coding agent 的 terminal-first TUI：主屏专注对话，工具调用折叠成一个持续刷新的 Build 块，细节按需进入详情页查看。

主要变化：

- 新增 `harness/tui_textual.py`：Textual + Rich 实现的默认 TUI。
- `main.py chat` 默认进入新版 Textual TUI；旧版无依赖 REPL 保留为 `--tui simple`。
- 顶部显示 `TINY AGENT` 彩色 ASCII logo，入口在 `harness/tui_textual.py::_tiny_agent_logo()`。
- `/` 命令菜单改为输入框上方的 inline command palette：输入 `/` 或 `/per` 时实时过滤，上下键选择，Enter 执行。
- 主消息流改为 transcript-first：用户输入、assistant 回复、thinking、Build 状态在同一条时间线里自然展示。
- 工具调用不再逐条铺满屏幕，而是聚合进 `BuildActivity`：
  - 运行中显示 spinner、模型、耗时、状态、toolcalls 数量和当前工具。
  - `assistant_delta.reasoning_content` 会渲染成 `Thinking:` 行；没有 reasoning 的模型不会伪造思考内容。
  - `tool_call/tool_result/tool_permission/tool_result_persisted/tool_context_modified` 都进入同一个 Build 详情。
- `Ctrl+O` 打开 Build details：
  - 左侧是真正可交互的工具列表 `OptionList`，支持 hover/click/highlight。
  - 右侧显示当前工具的 input、output、permission、persisted path、contextModifier 和 lifecycle。
  - 底部导航：`Parent esc`、`Prev left`、`Next right`。
- 滚动修复：transcript 不再每次刷新 `remove_children()`，改为固定 `TranscriptBody.update(...)`，避免 Textual 鼠标滚动/选择时撞上已删除节点。
- 滚动快捷键：`PageUp/PageDown/Ctrl+Home/Ctrl+End`；用户停在历史区时不会被新输出强行拉到底部。
- Markdown 正文继续走 Rich `Markdown` 渲染；工具 output 保持等宽文本，避免 ASCII tree / 日志 / 表格被误处理。

关键实现点：

- `ToolActivity`：按 `tool_call_id` 折叠单个工具完整生命周期。
- `BuildActivity`：一轮用户请求对应一个 OpenCode 风格 Build 块。
- `BuildDetailScreen`：Build 详情页，左侧工具列表 + 右侧详情 + 底部导航。
- `CommandMenuBody`：inline slash command 菜单。
- `TranscriptBody`：稳定 transcript surface，避免动态 mount/unmount 造成滚动崩溃。

## 2026-07-04：CH06 记忆系统 - 长期记忆底座与自动抽取

本阶段目标是参考 `CodeReference/claude-code-book/第二部分-核心系统篇/06-记忆系统-Agent的长期记忆.md`，把 TinyAgent 从“单次会话上下文”升级为有长期记忆的 coding agent。

CH06 参考结论：

- 记忆类型必须闭合为四类：
  - `user`：用户画像、背景、长期偏好
  - `feedback`：用户明确纠正或确认过的协作规则
  - `project`：不可从代码/git/文档直接推导的项目状态、决策背景、进行中目标
  - `reference`：外部系统入口、链接、频道、文档位置
- 不保存代码结构、Git 历史、当前临时任务、已在文档中的信息。
- 每条记忆是一个 Markdown 文件，带 frontmatter。
- `MEMORY.md` 是索引，不是记忆正文。
- 索引限制为 200 行 / 25KB。
- 记忆是线索，不是真相；行动前必须验证当前代码/文件状态。
- `autoMemoryDirectory` 不能信任 `projectSettings`，避免恶意仓库重定向写入敏感目录。

第一大步：确定性记忆底座

- 新增 `harness/memory.py`：
  - 四类型校验
  - frontmatter 读写
  - `MEMORY.md` 索引生成
  - index 200 行 / 25KB 截断
  - 默认项目隔离目录
  - trusted memory directory 解析
  - 忽略 `projectSettings.autoMemoryDirectory`
- 新增 `harness/memory_view.py`：
  - `/memory`
  - `/memory sources`
  - `/memory prompt`
  - `/memory list`
  - `/memory read`
  - `/memory tail`
  - `/memory status`
- 新增 `harness/tools/memory.py`：
  - `list_memories`
  - `read_memory`
  - `write_memory`
  - `forget_memory`
- system prompt 接入 memory index：
  - 有 `MEMORY.md` 时注入 `# Memory`
  - 明确提示“记忆是线索，不是真相”
  - 明确要求涉及当前代码/文件/函数/flag 时先验证
- TUI 接入：
  - 普通 TUI 支持 `/memory`
  - Textual TUI 支持 `/memory`
  - footer 显示 `memory:on/off`
  - 记忆状态文案统一使用 `◈`，例如 `[◈ Memory Saved]`
