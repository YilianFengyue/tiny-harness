# CH02 对话循环 — Agent 的心跳

## 2026-07-02 阶段总结：TinyAgent 心跳层升级完成

这一阶段的目标不是复刻 Claude Code，而是借鉴 `CodeReference/claude-code-book/第一部分-基础篇/02-对话循环-Agent的心跳.md` 里真正适合 TinyAgent 的核心设计，把原来的一次性任务 runner 升级成可持续交互、可观察、可恢复的 Agent loop。

主要参考点：

- CH02 的异步生成器思想：Agent loop 不应该是黑盒函数，而应该持续产出事件，让 UI/TUI 实时消费。
- CH02 的 `State / Continue / Terminal` 状态转换模型：每次继续循环要有明确原因，每次终止要有明确 reason。
- CH02 的完整 turn 生命周期：请求开始、模型响应、工具调用、工具结果回填、下一轮继续。
- CH02 的上下文预处理管线思想：上下文管理应从轻量预算到硬阻断逐级处理，而不是等 API 爆掉。
- CH02 的可观测性原则：trajectory 要能解释 Agent 每一步为什么发生。

本阶段已落地能力：

- 新增/强化事件协议：
  - `stream_request_start`
  - `assistant_delta`
  - `tool_start`
  - `tool_progress`
  - `tool_end`
  - `transition`
  - `context_edit`
- `run_start` 写入 `session_id`，方便区分同一 TUI session 下的多次 run。
- 保留 `Provider.complete(...)`，新增 `Provider.stream(...)`。
- `OpenAIChatProvider` 已支持 Chat Completions 真流式输出，并聚合成最终 `ModelTurn`，因此 loop 仍能保持原有工具协议和历史消息结构。

- Windows 下 bash 工具不再优先撞坏掉的 WSL launcher，会 fallback 到 PowerShell，修复了之前真实测试中 bash 不可用的问题。
- 上下文管理升级为 v2：

  - 超预算时清理旧 tool result。

- terminal reason 更细：
  - `aborted_streaming`
  - `aborted_tools`
  - `blocking_limit`
  - `prompt_too_long`
  - `model_error`

- 记忆类型必须闭合为四类：

## 2026-07-05：CH07 上下文系统 - 自动压缩 / 手动压缩 / TUI 水位

本阶段目标是参考 `CodeReference/claude-code-book/第二部分-核心系统篇/07`，把 TinyAgent 的上下文管理从“清理旧工具结果”升级成更接近长期 coding 会话的上下文系统：既能看见水位，也能在接近窗口上限时自动做摘要压缩，还能由用户在 TUI 中手动触发。

CH07 参考结论：

- 上下文不是简单 token 计数，而是一个运行时状态：安全、预警、自动压缩、阻断都要可见。
- 工具结果清理是便宜的 micro-compact；LLM 摘要压缩是更重但更保真的 summary-compact。
- 压缩不应该改写 trajectory 原始记录，只改写当前活跃 messages 视图。
- 摘要必须明确“这是线索，不是真相”，后续行动仍要验证当前文件和代码。
- TUI 需要在主交互区持续暴露上下文百分比，用户不应该等到 API 报错才知道上下文快满了。

已落地能力：

- 新增 `harness/compact.py`：
  - `compact_conversation(...)`
  - `CompactResult`
  - compact boundary message
  - compact summary message

## 2026-07-05：CH08 生命周期钩子与 Session 恢复 - Hook / Workspace History / Demo 验收

本阶段目标是参考 `CodeReference/claude-code-book/第二部分-核心系统篇/08`，补齐 coding agent 很关键的两类基础设施：一是可插拔的生命周期钩子，二是按工作区恢复历史 session。前者让外部脚本能在关键节点介入，后者解决“退出 TUI 再进来历史对话丢了”的问题。

CH08 参考结论：

- Hook 必须是结构化协议，不能只是随手跑脚本。

- `PreToolUse` 要能改写工具输入或阻断工具执行。
- `PostToolUse` 要能补充上下文或改写工具输出。
- `Stop` 可以追加上下文，让 agent 再进行一轮收尾。
- 工作区历史应该以文件夹为边界，而不是只存在进程内存里。

# 2026-07-05 CH10 Coordinator 第二大步

目标：

- 修正第一大步里“通过新建同类型 worker + 摘要上下文模拟续跑”的妥协。
- 实现真正 SendMessage / Stop-Resume：
  - worker 完成后保留原始 messages、provider、tool runtime、context manager。

本次实现：

- `BackgroundAgentRecord` 现在保存 `SubagentRuntime`：
  - worker messages 连续保留。
  - worker `ToolRuntimeState` 连续保留。
  - resume 次数写入 `resume_count`。
  - notification 标记每次完成后可重新 drain。
- `BackgroundAgentManager.send_message(...)`：
  - 只允许恢复已停止/已完成的 worker。
  - running worker 会返回可恢复错误。
  - unknown/non-resumable worker 会返回明确错误。
- `SubagentRuntime`：
  - 首次启动和恢复共用同一套运行现场。
  - 每次恢复产生新的 run trajectory。
  - worker identity 使用 background `agent_id`，trajectory `run_id` 每次运行独立。
- `agent` tool：
  - Coordinator 模式下新增 `agent_id` 参数。
  - `agent_id=null` 创建新 worker。
  - `agent_id=<task-id>` 对已有 worker 执行 SendMessage / Resume。
  - 普通 CH09 schema 不暴露 `agent_id`，避免影响默认模式。
- Coordinator prompt：
  - 明确两种模式：
    - CH09 fork/background：复制上下文开新 agent。
    - CH10 SendMessage：恢复同一个 stopped worker 的上下文和工具状态。
- TUI / viewer：
  - background start/done 事件增加 `resumed`、`resumable`、`resume_count` 显示字段。
